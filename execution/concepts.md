## prefill

我的解释：

- `prefill` 是模型处理输入 prompt 的阶段。它会把已有上下文一次性读入并计算出后续生成需要的 KV cache。
- 对用户来说，`prefill` 的影响通常会体现在首 token 出来之前，因此它和 `TTFT` 的关系更直接。

和现有经验的对应关系：

- 它更像系统里的“首次执行成本”或“初始化阶段成本”：不是稳定循环产出，而是为后续执行准备状态。

## decode

我的解释：

- `decode` 是模型逐 token 生成输出的阶段。每生成一个 token，都会基于已有 token 和 KV cache 继续推下一个 token。
- 它更像一个稳定循环过程，因此更适合用 `TPOT / ITL / output tok/s` 观察。

和现有经验的对应关系：

- 它更像 runtime 中的持续执行阶段：单步不一定很重，但会反复发生，所以单位产出效率和尾部抖动都很重要。

## KV cache

我的解释：

- `KV cache` 是推理过程中复用历史上下文计算结果的缓存。它减少重复计算，但会持续占用显存。
- 上下文变长、并发变高或输出变长时，KV cache 压力通常会变大。

和现有经验的对应关系：

- 它和已有的 cache / memory 经验很相通：复用能减少计算，但缓存容量、访存压力和资源占用会反过来约束吞吐与延迟。

## TTFT

我的解释：

- `TTFT` 是从请求发出到第一个 token 返回的时间。它更接近用户对“有没有快速开始响应”的感受。
- 当前 `serve` baseline 中，`Mean TTFT=169.71 ms`，可以作为第一条首包延迟基线。

和现有经验的对应关系：

- 它更接近系统服务里的首包时延或首次响应时间，通常会混合请求进入、调度、prefill 和首轮 decode 的成本。

## TPOT

我的解释：

- `TPOT` 是除首 token 外，后续每个输出 token 的平均生成时间。它更适合观察稳定生成阶段是否快、是否稳。
- 当前 `serve` baseline 中，`Mean TPOT=21.28 ms`，可以作为第一条 decode 阶段延迟基线。

和现有经验的对应关系：

- 它更接近稳定阶段的单位产出成本，类似“每处理一个单位数据要花多久”。

## 推理指标语言速记表

| 指标 | 我当前的理解 | 它主要反映什么 | 常见混淆点 |
| --- | --- | --- | --- |
| `TTFT` | 首 token 延迟 | 首包体验、prefill 与调度影响 | 不要和整体吞吐混为一谈 |
| `TPOT` | 后续 token 平均生成时间 | 稳定 decode 阶段速度 | 不要和单次请求总时长混为一谈 |
| `ITL` | 相邻 token 间隔 | 生成节奏是否平稳 | 不要只看均值，尾部抖动也重要 |
| `throughput` | 单位时间完成的请求数或 token 数 | 系统容量和整体效率 | 吞吐高不等于单请求体验一定好 |
| `memory` | 显存占用与峰值 | 模型权重、KV cache、runtime buffer 压力 | 显存占用和性能提升不总是正相关 |
